# Four-Man Team Executor Roadmap

This roadmap captures executor improvements to discuss and implement one step at a time.

Core rule: Four-Man Team must not duplicate state. Executor work is derived from the active task folder under `.4-man-team/tasks/<task-id>/`. Task-specific state stays inside that folder. Templates stay read-only. Orvo remains the only user-facing orchestrator.

## Context For Future Agents

Repository root:

- `20 My Projects/2020 AI/four-man-team`

Primary files:

- `src/prompts/execution.md` - Executor prompt and active-task behavior.
- `src/templates/EXECUTION.md` - task-local execution log template.
- `src/templates/STATUS.md` - task-local resume and checkpoint state.
- `src/templates/PLAN.md` - Arch output that Executor follows.
- `src/prompts/orvo.md` - Orvo checkpoint, routing, and resume behavior.
- `src/prompts/review.md` - Reviewer checks execution output and handoff quality.
- `tests/workflow/planner.test.ts` - prompt/template regression tests.
- `tests/workflow/four-man-team-cli.test.ts` - plan checker and task resolver tests.

Commands:

- `npm test` - full Bun test suite.
- `npm run task:resolve -- --root /path/to/project` - manually exercise active task resolution.
- `npm run plan:check -- /path/to/project/.4-man-team/tasks/task-id/PLAN.md` - manually exercise plan validation.

Current executor baseline:

- Resolves the active task by scanning the five most recent `.4-man-team/tasks/*/STATUS.md` files.
- Implements the active task's `TASK.md` and follows `PLAN.md` when present.
- Stops and reports when the plan is wrong instead of inventing a new architecture.
- Writes implementation notes to task-local `EXECUTION.md`.
- Records changed files, commands, blockers, skipped checks, and resume details.
- Updates `STATUS.md` before stopping when work is incomplete or files changed.

Local GSD references reviewed:

- `sample/gsd/agents/gsd-executor.md`
- `sample/gsd/get-shit-done/workflows/execute-phase.md`

## Local GSD Review Findings

Keep:

- Executor starts by loading project instructions and task context before edits.
- Executor distinguishes three execution patterns: fully automatic, checkpoint, and continuation.
- Deviation handling is rule-based: task-caused bugs, missing critical functionality, and blockers are fixed inline; architectural changes stop for a user decision.
- Package install failures are excluded from automatic blocker fixes because a failed package can be misspelled, hallucinated, or unsafe.
- Authentication failures are gates, not implementation failures.
- Human checkpoints return a structured handoff with completed work, blocker, awaited input, and continuation point.
- Analysis-paralysis guard stops endless reads before action.
- Self-check verifies claimed outputs before the executor hands off.
- Runtime/worktree safety matters when executor work is delegated or parallelized.

Adapt, do not copy:

- GSD uses `.planning/STATE.md`, `SUMMARY.md`, `ROADMAP.md`, `REQUIREMENTS.md`, and `gsd-sdk`; Four-Man Team should map those responsibilities into task-local `STATUS.md`, `EXECUTION.md`, `PLAN.md`, and Orvo routing.
- GSD commits each task; Four-Man Team V1 should keep commits under Orvo approval and repository policy.
- GSD spawns runtime-specific subagents; Four-Man Team should express the same safety rules in Orvo/Executor prompts and avoid assuming one runtime.
- GSD checkpoint continuation uses a fresh agent with explicit state; Four-Man Team can keep the same principle by requiring `STATUS.md` and `EXECUTION.md` to contain enough resume context for a new agent.

## GSD Patterns Worth Adopting

Useful ideas:

- clear execution decision paths for auto work, checkpoints, and continuation
- deviation rules that say when Executor may fix issues and when it must stop
- structured checkpoint returns with completed work, blocker, awaited input, and resume point
- authentication and setup gates treated as normal checkpoint flow, not failures
- package legitimacy checks before trying alternate package names
- anti-stall guard when an executor keeps reading without acting
- documentation lookup discipline for version-sensitive APIs
- project instruction enforcement before and during implementation
- runtime safety checks for delegated or parallel executor work
- final self-check before handoff to Reviewer

Patterns to avoid:

- no `.planning/` directory
- no `STATE.md`, `SUMMARY.md`, `ROADMAP.md`, or `REQUIREMENTS.md` runtime dependencies
- no GSD slash commands
- no `gsd-sdk` dependency
- no automatic per-task commit protocol in V1
- no global current-task pointer

## Step 1 - Executor Startup And Decision Flow [DONE]

Goal: make Executor startup and task selection predictable before edits begin.

What to add:

- Extend `src/prompts/execution.md` with an explicit decision flow:
  - read project instructions before implementation: `AGENTS.md`, `CLAUDE.md`, `GEMINI.md`, and any target-project equivalent when present
  - treat project instructions as hard constraints when they conflict with the plan
  - use official or local documentation for version-sensitive APIs instead of memory
  - if `PLAN.md` is absent, use `TASK.md` and record the missing plan in `EXECUTION.md`
  - if `PLAN.md` exists, follow the next incomplete task or wave
  - if task type is `auto`, execute and verify it
  - if task type starts with `checkpoint:`, stop and return a checkpoint to Orvo
  - if repo reality contradicts the plan, stop and report the mismatch
  - if resuming after a checkpoint, verify the checkpoint result before continuing
- Add an `## Startup Checks` section to `src/templates/EXECUTION.md`:
  - instructions read
  - relevant docs consulted or reason not needed
  - active task resolved
  - next task selected
- Keep current active-task resolution unchanged.
- Keep all progress in `EXECUTION.md` and `STATUS.md`.

Why this fits Four-Man Team:

- It makes the existing executor prompt stricter without adding another workflow.
- It preserves Orvo as the user-facing router.
- It uses task-local files as the only resume source.

Implementation references:

- Update `src/prompts/execution.md`.
- Update `src/templates/EXECUTION.md`.
- Add or extend a regression test for executor prompt/template wording.

Acceptance criteria:

- `npm test` passes.
- Executor prompt distinguishes startup checks, auto work, checkpoint tasks, plan mismatch, and resume.
- `EXECUTION.md` records which project instructions and docs governed execution.
- No `.planning/` or global state file is introduced.

## Step 2 - Deviation Rules [DONE]

Goal: let Executor fix task-local problems without expanding scope.

What to add:

- Add bounded deviation rules to `src/prompts/execution.md`.
- Executor may fix inline:
  - bugs caused by current task changes
  - missing correctness or security basics required for the current task
  - blockers that prevent the current task from completing
- Project-instruction-driven fixes, such as mandated validation, path handling, or security rules, count as missing critical functionality when required for the current task.
- Executor must stop and ask Orvo for:
  - architecture changes
  - product decisions
  - new infrastructure
  - dependency or framework swaps
  - changes outside approved scope
- Executor must log each deviation in `EXECUTION.md`.
- After three failed fix attempts on one task, stop fixing and report the blocker to Orvo.
- Out-of-scope discoveries go to `STATUS.md` `Known Gaps`, not a new deferred-items file.

Why this fits Four-Man Team:

- It adapts GSD's useful auto-fix boundary without importing GSD's summary/state files.
- Reviewer can audit deviations from `EXECUTION.md`.

Implementation references:

- Update `src/prompts/execution.md`.
- Update `src/templates/EXECUTION.md` to make deviation entries easier to scan.
- Update `src/prompts/review.md` so Reviewer checks deviation scope.

Acceptance criteria:

- Executor prompt has explicit auto-fix and stop rules.
- `EXECUTION.md` records deviations, attempts, rule category, and Orvo approval when needed.
- `STATUS.md` records out-of-scope gaps without new global state.
- Reviewer prompt treats unsafe deviations as review findings.

## Step 3 - Structured Checkpoint Return

Goal: make human pauses precise and resumable.

What to add:

- Add a fixed checkpoint shape to `src/templates/EXECUTION.md` and executor prompt:

```markdown
## CHECKPOINT REACHED

**Type:** [decision / human-verify / human-action / external-setup]
**Progress:** [completed]/[total] tasks complete
**Current task:** [task number and name]
**Status:** [blocked / awaiting verification / awaiting decision / awaiting setup]
**Blocked by:** [specific blocker]

### Completed Work

- [Task] - [files changed] - [verification result]

### Awaiting

[Exact thing Orvo must ask the user to decide, verify, provide, or do.]

### Resume After

[Concrete next step Executor should run after Orvo records the answer in STATUS.md.]
```

- Orvo records the checkpoint result in `STATUS.md`.
- Executor resumes only after `STATUS.md` contains the needed answer or confirmation.
- Continuation must not redo completed work; it verifies prior completed tasks from `STATUS.md` and `EXECUTION.md`, then starts at the resume point.

Why this fits Four-Man Team:

- It replaces ad hoc stopping messages with a task-local handoff.
- It keeps checkpoint ownership with Orvo.
- It adapts GSD's "fresh continuation agent with explicit state" pattern without requiring GSD state files.

Implementation references:

- Update `src/prompts/execution.md`.
- Update `src/templates/EXECUTION.md`.
- Update `src/prompts/orvo.md` only if Orvo needs exact checkpoint-recording wording.

Acceptance criteria:

- Checkpoint output includes completed work, blocker, awaited input, and resume step.
- Checkpoint result lives in `STATUS.md`.
- Resume instructions identify exactly which task or wave continues next.
- No separate checkpoint tracker is added.

## Step 4 - Auth And External Setup Gates

Goal: treat unavailable credentials and user-only setup as expected checkpoint flow.

What to add:

- Executor recognizes auth/setup indicators such as:
  - `not authenticated`
  - `not logged in`
  - `unauthorized`
  - `401`
  - `403`
  - missing required environment variable
  - CLI asks the user to log in
- Executor stops with `checkpoint:human-action` or `checkpoint:external-setup`.
- Executor provides exact verification command or condition after setup.
- Executor never prints, stores, or commits secret values.
- Before a `checkpoint:human-verify`, Executor ensures the verification environment is ready when practical, such as starting the local server or running required setup commands.
- User-only actions should stay rare: the agent handles CLI/API/browser automation when it can do so safely.

Why this fits Four-Man Team:

- It matches the existing checkpoint model.
- It avoids treating user-only setup as a failed implementation.

Implementation references:

- Update `src/prompts/execution.md`.
- Update `src/prompts/orvo.md` for safe secret handling if not already explicit enough.
- Update `src/templates/EXECUTION.md` checkpoint section.

Acceptance criteria:

- Auth/setup gates are recorded as checkpoint flow, not generic blockers.
- Secret values are never written into tracked task files.
- Resume instructions include the post-setup verification step.
- Human verification checkpoints include URLs, expected behavior, or concrete visual/functional checks, not vague "check it" wording.

## Step 5 - Package Legitimacy Guard

Goal: prevent unsafe package substitution when installs fail.

What to add:

- If a package install fails or a package cannot be found:
  - do not install a similarly named package
  - do not retry with guessed package names
  - stop with `checkpoint:human-verify`
  - ask Orvo to have the user verify the package name and legitimacy
- Record the package name, command, failure output summary, and required user decision.
- Mark package-legitimacy checkpoints as blocking human checkpoints that cannot be auto-approved later.

Why this fits Four-Man Team:

- It is a compact safety rule with high value.
- It does not require new tooling.

Implementation references:

- Update `src/prompts/execution.md`.
- Optionally add package legitimacy wording to checkpoint rules in `src/templates/PLAN.md` later.

Acceptance criteria:

- Executor prompt forbids package-name guessing.
- Failed install produces a human verification checkpoint.
- Package checkpoints cannot be auto-approved by any future auto mode.
- Reviewer can flag unsafe dependency substitution.

## Step 6 - Anti-Stall And Runtime Safety

Goal: stop Executor from burning context on endless inspection and avoid runtime delegation traps.

What to add:

- Add a simple executor guard:
  - after five consecutive read/search operations without edit, command execution, or checkpoint output, Executor must stop and state why it has not acted
  - then either perform the next concrete action or report the exact blocker
- Treat this as prompt behavior, not CLI enforcement for now.
- Add delegated-execution safety guidance:
  - if an executor agent is spawned in the background, Orvo should not duplicate that work locally
  - if multiple executors may run in parallel, plans must avoid overlapping file ownership
  - when worktree isolation is unavailable or unsafe, run executor work sequentially
  - do not rely on runtime-specific completion callbacks; verify via task files and git state when needed
- Explicitly avoid GSD worktree cleanup machinery in V1; Four-Man Team should only add worktree rules if Superset integration actually uses worktrees.

Why this fits Four-Man Team:

- It improves execution momentum without adding infrastructure.
- It gives Orvo a clear signal when context is missing.
- It adapts the important runtime lessons from GSD while staying Superset-first.

Implementation references:

- Update `src/prompts/execution.md`.
- Update `src/prompts/orvo.md` if Orvo later spawns parallel executors.
- Update `src/templates/PLAN.md` only if file-ownership rules need stronger wording.

Acceptance criteria:

- Executor prompt includes the anti-stall rule.
- Blocked reports identify the missing fact, file, permission, or decision.
- Orvo prompt prevents duplicate local work while delegated executor work is active.
- Parallel execution is only allowed when file ownership is disjoint.

## Step 7 - Executor Self-Check

Goal: make ready-for-review claims auditable.

What to add:

- Before setting `Ready for Review: YES`, Executor must verify:
  - changed files listed in `EXECUTION.md` exist
  - commands listed in `Verify:` were run or explicitly skipped with reason
  - `STATUS.md` has current `Next action`, `Last completed action`, `Handoff note`, and `Files Touched This Session`
  - deviations and skipped checks are recorded
  - no placeholder text remains in the execution log
- Executor also checks for new stubs or placeholders introduced in changed files when those would block the task goal.
- Executor records residual risks or known stubs in `EXECUTION.md` and `STATUS.md` instead of marking them complete silently.
- Add a `## Self-Check` section to `src/templates/EXECUTION.md`.

Why this fits Four-Man Team:

- It adapts GSD's self-check idea into task-local logs.
- Reviewer gets a concrete checklist.

Implementation references:

- Update `src/templates/EXECUTION.md`.
- Update `src/prompts/execution.md`.
- Update `src/prompts/review.md` so Reviewer checks self-check claims.

Acceptance criteria:

- `EXECUTION.md` has a self-check section.
- Executor cannot mark ready for review without verification status and handoff fields.
- Self-check fails when claimed changed files, verification evidence, or handoff fields are missing.
- Reviewer prompt includes stale or false self-checks as findings.

## Step 8 - Optional Future Commit Protocol

Goal: decide later whether Executor should own commits.

Decision for now:

- Do not adopt GSD's automatic per-task commit protocol in V1.
- Keep commit/merge behavior under Orvo approval and repository policy.

Possible future version:

- Add config-gated commit behavior only if:
  - Orvo asks before committing
  - Executor stages only task-owned files
  - dirty worktree handling is explicit
  - rollback notes are required
  - Reviewer checks commit scope
- If worktrees are ever added, include path containment, branch safety, and no-broad-cleanup rules before allowing executor commits.

Why this fits Four-Man Team:

- Current project scope says V1 is prompts, config, and handoff templates.
- Automatic commits are higher-risk than prompt/runtime handoff improvements.

Acceptance criteria:

- No commit automation is added by default.
- Any future commit protocol is opt-in and Orvo-controlled.
